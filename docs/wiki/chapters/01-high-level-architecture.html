01. System Architecture

Graph-Native Orchestration & Industrial Control Flow

On this page

Core Philosophy

The Factory Agent is not a linear chain of prompts. It is a state-driven orchestration engine built on LangGraph. The architecture follows a "Plan-First, Verify-Always" approach, ensuring that every industrial action is semantically validated and approved before execution.

Key Tenet: Separation of Concerns

The system separates Planning (reasoning about tools), Execution (making HTTP calls), and Validation (checking if results match requirements) into distinct, isolated graph nodes.

Graph Topology

The execution flow is managed by a Directed Acyclic Graph (with controlled cycles for repair). Below is the high-level node structure defined in builder.py:

graph TD
    Input[input_layer] --> Split[intent_splitter]
    Split --> Prep[prepare]
    Prep --> Planner[planner]
    
    Planner -->|Clarification| ClarifyEnd[clarify_end]
    Planner -->|Staged Action| Guard[decision_guard]
    Planner -->|Final Result| Synth[synthesize_plan]
    Planner -->|Retry| Planner
    
    Guard -->|Wait/Refine| Planner
    Guard -->|Execute| ToolExec[tool_execution]
    
    ToolExec -->|Fatal Error| FatalEnd[fatal_end]
    ToolExec -->|Success| RelFilter[relevance_filter]
    
    RelFilter -->|Retry Planning| Planner
    RelFilter -->|Move to Plan| Synth
    
    Synth --> FinalVal[final_validator]
    
    FinalVal -->|Refine| Planner
    FinalVal -->|Dry Run| DryRun[bundle_dry_run]
    FinalVal -->|Commit| Commit[commit]
    FinalVal -->|Finish| END
    
    DryRun --> FinalVal
    Commit --> END
    FatalEnd --> END
    ClarifyEnd --> END

Node Responsibilities

Intent Splitter

Decomposes complex user queries into planner-owned v2 agenda items. If a user asks to "fix machine A and update schedule B", the loop records distinct requirements, evidence, and response obligations for each sub-problem.

Decision Guard

A safety layer that inspects planned ToolCalls. It enforces constraints (e.g., "cannot delete production records on weekends") before allowing the tool_execution node to fire.

Relevance Filter

Post-execution audit. It checks if the tool output actually helped solve the current intent. It prevents the planner from getting stuck in "hallucination loops" with irrelevant data.

Final Validator

The gatekeeper of the PlanDraft. It ensures the proposed answer is supported by "evidence" collected during tool execution.

State Management (AgentState)

The "brain" of the agent is the AgentState TypedDict. Unlike typical LLM memory, this state is structured and uses Reducers to manage how data accumulates.

Field Type Update Logic
messages list[AnyMessage] add_messages (LangGraph native delta accumulation)
staged_writes list[dict] replaceable_list_reducer (Allows clearing stale writes during repair)
intents list[dict] operator.add (Append-only trace of all split intents)
tool_outputs list[dict] operator.add (Full history of every HTTP interaction)

Two-Phase Commit Lifecycle

To prevent accidental industrial damage, the system implements a strict two-phase commit protocol for any "Write" actions (POST/PUT/PATCH/DELETE):

  1. Phase 1: Dry Run - Tools are executed with a dry_run=true flag (if supported) or staged in staged_writes while the system simulates the outcome.
  2. Phase 2: HITL Approval - The graph hits an Interrupt. The frontend receives an approval_required event via SSE.
  3. Phase 3: Commitment - Upon human approval, the graph resumes at the commit node, which executes the real API calls using Idempotency Keys to prevent double-execution.

Supporting Subsystems

Dynamic RAG Pipeline

The system uses a Hybrid Retrieval (BM25 + Vector) strategy to inject real-time machine manuals and maintenance logs into the LLM context. It includes a Cross-Encoder Reranker to ensure only the top 3 most relevant documents are passed to the planner.

Tool Registry

A centralized hub that manages over 100+ industrial tools. It handles automatic Pydantic schema generation from OpenAPI specs, enabling the agent to "know" how to call the Go backend without hardcoded logic.

Technical Stack

The architecture bridges high-level Python reasoning with low-level Go execution:

  • Reasoning: Python 3.11+, LangGraph, LangChain, Pydantic V2.
  • Interface: FastAPI (Uvicorn) with Server-Sent Events (SSE).
  • Backend: Industrial Go API (eMAS Core).
  • Persistence: SQLite (Local) / PostgreSQL (Prod) for graph checkpointing.
