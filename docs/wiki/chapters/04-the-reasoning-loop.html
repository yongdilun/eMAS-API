04. The Reasoning Loop

Recursive Planning & Deterministic Recovery

On this page

Overview

The **Reasoning Loop** is the cognitive center of the Factory Agent. It iterates through perception, planning, and validation until the current intent is achieved. While the LLM provides flexibility, the loop is governed by rigid limits and fallback mechanisms to ensure industrial stability.

The Planner Node

The planner_node is responsible for evaluating the AgentState and the current working_intent to determine the next action. It operates in a stateless manner relative to the LLM, relying entirely on the provided history and context.

  • Dynamic Context: The prompt includes only the relevant tool schema and the recent tool_outputs to save tokens and focus reasoning.
  • Recursion Control: The graph enforces a recursion_limit to prevent infinite planning loops in ambiguous scenarios.

The Decision Object

Every reasoning step results in a structured PlannerDecision. This ensures that the agent's output is always machine-parseable and verifiable.

Field Purpose
kind The action type, such as domain_tool, request_approval, clarify, or finalize.
tool_calls A list of specific API calls with their resolved arguments.
risk_level Safety classification used by the decision_guard to trigger interrupts.
decision_summary Natural language rationale for the chosen path (used in SSE streaming).

Deterministic Plan Repair

To survive LLM hallucinations or JSON parsing failures, the system implements a Deterministic Repair mechanism.

If the Planner fails to generate a valid decision or hits the iteration cap, the system invokes _deterministic_plan_repair.

This "Plan B" uses hard-coded logic to attempt a safe completion of the intent (e.g., stopping a machine if the intent was "stop") without relying on further LLM inference.
